# 访谈纪要融合流程说明（自然语言版）

## 1. 我在解决什么问题
这个流程要解决的是：把 `转写.docx` 里的访谈对话，准确地整合进 `访谈纪要.docx` 的每个“回复：”位置，最终得到一份可交付的 `访谈纪要完成版ClaudeQwen.docx`。  
同时需要满足三个关键要求：  
1. 语义匹配优先，不做生硬关键词拼接。  
2. 新增内容标红，旧内容只把“改动片段”标红。  
3. 低置信度条目不拦截流程，但要在回复后加提示注释。  

## 2. 整体思路
我的思路是“结构定位 + 语义匹配 + 规则兜底 + 精准回写”四段式：
1. 先在纪要里把每个“回复槽位”精确定位出来。  
2. 再让模型在转写里为每个问题找最合适的证据和答案。  
3. 用统一阈值规则做质量兜底，保证输出行为稳定。  
4. 最后把结果写回原始 docx，并保留版式、执行红字标注。  

## 3. 输入资源与中间资源
输入有两份：`转写.docx` 和 `访谈纪要.docx`。  
流程会生成中间文件（例如 `temp_transcript.md`、`temp_summary.md`、`questions.json`、`transcript_chunks.json`、`llm_input.json`），它们的作用是让匹配过程可追踪、可复核，不是最终交付件。  

## 4. 关键步骤详解

### 步骤 A：从纪要中识别“问题-回复槽位”
我会遍历纪要文档，把每个“回复：”都识别成一个待填位置。  
然后从该位置向上回溯，跳过章节标题，找到最近的“问题句”（优先含问号）。  
这一步最终形成一个结构化映射：`question_id -> 问题文本 -> 回复段落索引`。  
这个映射是后续回写的定位基础，确保答案写到正确位置。  

### 步骤 B：把转写拆成可匹配证据块
我会把转写按段拆分成 `chunk`，每个 `chunk` 都有唯一 `chunk_id`，并尽量保留说话人、时间戳和文本。  
这样做是为了让后面的回答有证据可追溯，而不是“凭感觉总结”。  

### 步骤 C：语义匹配与答案生成
模型接收两类信息：  
1. 问题列表（来自纪要）；  
2. 对话切块（来自转写）。  

然后按规则逐题处理：  
1. 先判断哪些对话片段在语义上真正回答了该问题。  
2. 如果同一问题有分散回答，合并成 2-4 句精炼回复。  
3. 如果转写一次提问覆盖纪要多个问题，先拆分回答要点，再分别映射到对应问题。  
4. 对同题多回复槽位，按槽位语义做差异化落位，避免机械重复。  
5. 必须给出证据片段 ID。  
6. 找不到明确回答则置空，不强行填。  
7. 若识别到纪要外临时问题，可输出为 `new_items`。  

### 步骤 D：冲突裁决和低置信度处理
如果候选回答冲突，按固定顺序裁决：  
1. 优先直接回答核心意图的内容；  
2. 再看完整性和可执行性；  
3. 可互补则合并；  
4. 若有后续明确修正，优先后者；  
5. 仍冲突时选更具体、事实细节更多者。  

然后给出置信度并走统一阈值：  
1. `>=75`：写入。  
2. `55-74`：写入并追加 `（注：低置信度，需要人工审核）`。  
3. `<55`：留空。  

## 5. 回写 docx 的方式
回写不是重建新模板，而是直接编辑原纪要文档中的“回复段落”。  
这样可以最大程度保留原有版式、字体、段落结构和表格。  
在颜色处理上执行两条：
1. 新增文本标红。  
2. 原有文本若有修订，只把改动片段标红，不把整段全红。  
另外，若存在 `new_items`，会在“资料清单”前插入“临时新增访谈问题”补充块，新增问题与回复均标红。  

## 6. 为什么要用 Agent-First 流程
考虑到你希望这套方法长期给 Codex / Claude Code 使用，我把流程设计成“任务包驱动”：  
1. 先把问题分批，生成 `Bxxx.input.json`。  
2. 让 Agent 逐批生成 `Bxxx.output.json`。  
3. 最后统一合并并回写。  

这样做的好处是：  
1. 批量任务稳定，不容易超上下文。  
2. 每批结果可独立追踪和重跑。  
3. 同一套规则可复用到很多项目。  

## 7. 最终交付与可复用性
最终交付是 `访谈纪要完成版ClaudeQwen.docx`。  
这套流程的核心是“结构定位 + 语义匹配 + 阈值规则 + 精准回写”，只要输入文件格式类似，后续项目可以直接复用。  
如果以后你需要，我还可以在这份流程上继续扩展质量报告（例如每题证据摘要、低置信度列表、未匹配原因统计）。
