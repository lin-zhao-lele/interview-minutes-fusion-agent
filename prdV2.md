# PRD V2：访谈纪要自动补全与修订

## 1. 背景与目标
- 将音频转写内容（`转写.docx`）中的有效问答信息，准确整合到访谈纪要（`访谈纪要.docx`）对应问题下的“回复：”位置。
- 产出一份可交付文档：`访谈纪要完成版ClaudeQwen.docx`。
- 在不破坏原文档排版与风格的前提下，完成补全、修订和标注。

## 2. 输入与输出
### 2.1 输入文件
- `转写.docx`：音频稿，包含访谈对话原始问答。
- `访谈纪要.docx`：待补全和修订的纪要模板。

### 2.2 中间文件（用于分析，不作为最终回写源）
- `temp_transcript.md`：由 `转写.docx` 转换得到，用于问答抽取与匹配。
- `temp_summary.md`：由 `访谈纪要.docx` 转换得到，用于结构化比对与校核。

### 2.3 输出文件
- `访谈纪要完成版ClaudeQwen.docx`：最终交付件。

## 3. 范围与任务
- 理解 `转写.docx` 中的对话语义，识别问题与回答关系。
- 将回答映射到 `访谈纪要.docx` 对应问题后的“回复：”位置。
- 重点处理纪要中从第 4 页开始的大量“回复：”空白内容。
- 核对并修正纪要中已有回复，使其与转写内容一致、表达更清晰。

## 4. 已确认业务规则（本轮共识）
- 规则 1：新增内容与改动过的旧内容都要标红。
- 规则 2：若同一问题在音频中有多段分散回答，需要合并为一段精炼回复。
- 规则 3：若在音频中找不到明确对应回答，该问题“回复：”留空，不补“未提及”等占位文字。
- 规则 4：旧文本修订时，仅对改动片段标红，不对整句整段整体标红。
- 规则 5：访谈纪要中的多个问题可对应转写中的一次提问/一次回答，需要先拆解要点再分别填充。
- 规则 6：转写中若存在纪要未预置的临时新增问题，需要连同对应回复写回纪要。

## 5. 文档处理策略（关键实现约束）
- `docx -> md` 转换是必要的分析手段，但不是唯一工作底稿。
- 最终回写必须直接操作 `docx`，不能依赖从 `md` 反向生成来替代，以避免样式、段落、编号、表格和 run 级格式丢失。
- `temp_summary.md` 仅用于：
  - 对齐问题结构；
  - 追踪补全进度；
  - 做内容核对与复核。

## 6. 编辑与质量要求
- 保持原文风格、语气和纪要表达习惯，不引入与原语境冲突的表述。
- 精炼优先：在保证事实完整的前提下，避免口语化重复与冗余；单个问题回复以几句为宜。
- 忠实优先：不得编造事实，不得填入无来源信息。
- 标注一致：所有新增和被修改的既有文字均使用红色字体标注；对既有内容采用“仅改动片段标红”。

## 7. 匹配策略（新增）
- 以语义匹配为第一优先级，判断回答是否真实回应该问题的核心意图。
- 关键词匹配作为辅助证据，不作为主判定依据。
- 若关键词相关但语义不构成直接回答，不强行填充，按“留空”规则处理。
- 多对一映射策略（纪要多问题 <- 转写一次提问）：
  - 先将转写回答拆分为原子要点（如主体信息、规模信息、人员信息）。
  - 再按纪要问题意图将要点分别分配到不同问题，避免多个问题直接复制同一答案。
  - 若某问题缺少独立证据支撑，优先留空而非复用弱相关内容。
- 临时新增问题策略：
  - 识别转写中存在、但纪要中无对应题目的问答。
  - 满足证据约束时，作为新增条目写回纪要。
  - 新增条目包含：问题文本、回复、证据与置信度信息。
- 语义冲突裁决规则（按顺序执行）：
  - 先选直接回应该问题核心意图的候选答案。
  - 在直接候选中优先信息更完整、可执行性更高的版本。
  - 多个候选互补时，合并为 2-4 句精炼回复。
  - 若说法冲突，优先采用同一受访方后续明确修正的表述。
  - 若仍冲突，优先采用更具体、事实细节更多（时间/金额/对象/动作）的表述。
  - 若最终仍无法判定，按“留空”规则处理。

## 8. 自动化处理流程（新增）
- 采用全自动三阶段处理，但不设置人工复核关卡：
  - 阶段 1：问题-答案语义匹配，输出结构化结果（`question_id/status/confidence/evidence/answer_draft`）。
  - 阶段 2：基于证据进行 2-4 句精炼改写，并执行事实一致性检查。
  - 阶段 3：自动回写 `docx` 并完成红色标注。
- 低置信度或冲突未完全消解的条目，不中断自动流程；在对应“回复：”正文后追加提示：
  - `（注：低置信度，需要人工审核）`
- 证据约束：
  - 每条回复必须可追溯到转写证据片段。
  - 无证据不填充。
  - 先抽取证据与候选答案，再进行精炼改写。
- 新增问题写回：
  - 输出中允许 `new_items`（临时新增问题列表）。
  - `new_items` 仅保留 status=filled 且有证据的条目。
  - 回写位置为纪要“其他问题”补充区（当前实现为“资料清单”前插入补充块）。
- 低置信度判定采用“硬规则 + 量化评分”：
  - 硬规则（任一命中则留空）：
    - 无可追溯证据片段。
    - 语义不直接回答问题核心意图（仅相关但未回答）。
    - 存在明显冲突且无法按裁决顺序消解。
  - 量化评分（0-100）：
    - `S` 语义匹配度（35%）
    - `E` 证据覆盖度（25%）
    - `C` 一致性（20%）
    - `F` 完整性（10%）
    - `T` 可追溯性（10%）
    - 公式：`confidence = 0.35S + 0.25E + 0.20C + 0.10F + 0.10T`
  - 阈值策略：
    - `confidence >= 75`：正常写入（2-4 句）。
    - `55 <= confidence < 75`：写入并追加 `（注：低置信度，需要人工审核）`。
    - `confidence < 55`：留空。
  - 结构化输出字段：
    - `question_id`
    - `status`（`filled`/`blank`）
    - `confidence`
    - `evidence`
    - `answer_draft`
    - `note_required`（`true`/`false`）
    - `new_items`（可选）：
      - `question_text`
      - `status`（`filled`/`blank`）
      - `confidence`
      - `evidence`
      - `answer_draft`
      - `note_required`（`true`/`false`）

## 9. 验收标准
- 每个可匹配问题均有准确回复，并落在正确“回复：”位置。
- 无法匹配的问题保持空白。
- 临时新增问题若有明确证据，需写入纪要并完成红字标注。
- 原有版式、段落结构、编号、表格、字体体系保持可用且无明显破坏。
- 输出文件名与路径正确：`访谈纪要完成版ClaudeQwen.docx`。
- 低置信度条目均在对应回复后附加 `（注：低置信度，需要人工审核）`。

## 10. 当前阶段边界
- 当前阶段先进行方案讨论与规则确认。
- 暂不执行最终文档生成与回写。
